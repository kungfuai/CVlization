# Miles RL Experiments for OMR

**Previous**: [`2026-05-05-rl-experiments.md`](2026-05-05-rl-experiments.md) — TRL/unsloth GRPO findings.

## Summary

Attempted to use Miles (RL framework with decoupled SGLang inference) to
train Qwen3.5-9B-VLM with GRPO on synthetic Level 9 data. After fixing
5+ infrastructure issues, we got rollouts generating successfully but
hit a Megatron-LM kernel limitation (GDN doesn't support packed sequences).

**Conclusion: Miles + Qwen3.5-VL is not production-ready** as of
2026-05-11. Multiple open community bugs (slime #1815, #1713, #1894)
confirm the VL variant has unresolved issues. Text-only Qwen3.5 works in
Miles, but our use case requires vision.

Returning to TRL/unsloth GRPO for further experiments.

## Why we tried Miles

After 6 TRL GRPO runs (v1-v6b), we plateaued at 84% on Level 9 (matching
SFT baseline). Miles offered:

1. **Parallel rollouts** via SGLang (TRL was sequential)
2. **True on-policy training** (bit-wise identical log-probs)
3. **Full fine-tuning** instead of LoRA r=32 (capacity ceiling)
4. **First-class VLM support** — advertised for Qwen3-VL family
5. **Multi-GPU** (TRL ran single-GPU in our setup)

## Setup

- Model: Qwen3.5-9B-VLM (merged SFT, `outputs/tamqjf4k_merged`)
- Data: Level 9 synthetic, 20 samples (validation), 3115 (full)
- Reward: `SequenceMatcher.ratio()` on MXC2 pitches (matches eval metric)
- Backend: Megatron (FSDP deprecated for SGLang v0.5.10)
- Code: `examples/perception/optical_music_recognition/vlm_omr_miles/`

## Issues encountered (in order)

### Issue 1: FSDP deprecated
Miles raises `ValueError: FSDP backend has known issues with SGLang v0.5.10`.
**Fix**: switch to Megatron backend with bridge mode (geo3k_vlm default).

### Issue 2: Local model path rejected by snapshot_download
`HFValidationError: Repo id must be in the form 'repo_name'`.
**Fix**: detect absolute path in `download_model()`, skip HF download.

### Issue 3: Megatron-LM not on PYTHONPATH
`ModuleNotFoundError: No module named 'megatron.training'`.
**Fix**: add `/root/Megatron-LM` to PYTHONPATH in train.sh.

### Issue 4: MTP layer name mismatch (slime #1894)
`AttributeError: 'NoneType' object has no attribute 'megatron_module'`
during HF→Megatron weight conversion.

Megatron-LM upstream renamed `transformer_layer` → `mtp_model_layer`
in MTP submodule. Megatron-bridge's `qwen35_vl_bridge.py` registers
mappings for the new name, but our Megatron-LM still emits the old
name in param names.

**Fix**: text-replace `mtp_model_layer` → `transformer_layer` (20 sites)
in megatron-bridge VL bridge file. See `qwen3_5_mtp_patch.py`.

Upstream fix in megatron-bridge PR #3769 (unmerged as of 2026-05-11).

### Issue 5: Custom RM path format
`ModuleNotFoundError: No module named '/workspace/reward'`.
Miles loads custom reward via `importlib.import_module(module_path)`,
not file path.

**Fix**: pass `reward.reward_fn` and add `/workspace` to PYTHONPATH.

### Issue 6 (blocking): GDN + packed_seq incompatibility
`NotImplementedError: GDN does not support packed sequence for now.`
in `Megatron-LM/megatron/core/ssm/gated_delta_net.py:300`.

Qwen3.5 uses Gated Delta Net (GDN) attention. Miles forces
`args.variable_seq_lengths = True` for Megatron backend, which
creates `packed_seq_params`, which GDN's forward pass explicitly
rejects. This is a kernel-level issue, not a config option.

**No fix attempted** — would require either patching Miles to not
force varlen, or patching the GDN kernel to handle packed sequences.
Both are invasive and risk correctness.

## What worked end-to-end (validation output)

Despite the final blocker, we verified:

- ✅ Megatron model loaded via bridge (4.8B params per TP rank)
- ✅ SGLang VLM engine started with TP=2
- ✅ Both GPUs initialized
- ✅ Reward function imported via `module.attr` path
- ✅ Rollout generated (`POST /generate HTTP/1.1 200 OK`)
- ✅ Performance metrics: 19.5s per rollout, 105 tokens/sec/GPU
- ✅ Reward computed (returned -1.0 because model output was raw XML)

So the pipeline is ~80% functional. The blocker is purely in the
training step's attention kernel.

## Lessons learned

### About RL framework selection
1. **Verify VLM support before investing**. The text path is much
   better tested. Don't assume "Qwen3-VL supported" means
   "Qwen3.5-VL supported".
2. **Check open issues for your exact model/version combo** before
   starting. slime #1815, #1894, #1713 all directly affected us
   and were findable in 5 minutes of search.
3. **Bleeding-edge frameworks have a Day-N tax**. radixark/miles
   was created 2 days before slime #1894 was filed — we were the
   second team to hit the bug.

### About debugging RL infra
1. **Compatibility tests first** — `test_sglang_load.py` correctly
   verified mbridge import + model load before the harder bugs.
   But it didn't verify the actual *training step* path, where
   issue #6 lives.
2. **Layered fixes reveal underlying assumptions**. Each fix
   unblocked the next layer (model load → rollout → reward → train).
   Worth budgeting for: each layer can have its own bug.

### About SFT model architecture
1. **Vision LoRA is unnecessary** (we proved 97% on Level 7a with
   `finetune_vision_layers=False`). But our SFT used vision LoRA,
   creating a VLM checkpoint that triggers the harder code paths
   in Miles.
2. **For future SFT runs**: use `finetune_vision_layers=False` AND
   start from the text-only Qwen3.5-9B if planning to use Miles
   later. Our merged VLM model is what makes Miles unstable.

### About the broader RL plateau
Even if Miles worked, it's unclear if it would break the 84% Level 9
ceiling. The TRL experiments suggested:
- More generations helped (37% → 8% zero-variance with 8 gens)
- But accuracy didn't move (84% → 84%)
- The errors may be systematic rather than addressable by RL

Miles' theoretical advantages (true on-policy, more parallelism, full
fine-tune) might break the plateau, but we don't have evidence either
way for our specific task.
